# SPDD REASONS Canvas Generator

Используй этот скилл для создания структурированного REASONS Canvas перед генерацией кода.

## Инструкция по использованию

Когда тебе дают требование или задачу, сначала создай REASONS Canvas по следующей структуре:

## REASONS Canvas Шаблон

```markdown
# REASONS Canvas: [Название задачи]

## R - Requirements
**Цель:** Четкое описание проблемы, которую нужно решить
**Функциональные требования:**
- [ ] Требование 1
- [ ] Требование 2

**Definition of Done (DoD):**
- [ ] Критерий 1
- [ ] Критерий 2

**Принятия критериев:**
- Что считается успешным завершением

## E - Entities
**Доменные сущности:**

### [Entity Name]
- Свойства:
  - `property: type` — описание
- Отношения:
  - Связь с другими сущностями

**Схема данных:**
```
[ER-диаграмма или описание структуры данных]
```

## A - Approach
**Стратегия решения:**

1. **Архитектурный подход:**
   - Паттерны проектирования
   - Выбранные технологии и обоснование

2. **Технические решения:**
   - Почему выбрано это решение
   - Альтернативы и почему от них отказались

3. **Поток данных:**
   - Как данные перемещаются через систему

## S - Structure
**Контекст изменения в системе:**

### Компоненты
```
src/
├── [component]/
│   ├── index.ts
│   ├── [component].ts
│   └── [component].test.ts
```

### Зависимости
- Существующие модули: []
- Новые зависимости: []
- API контракты: []

### Точки интеграции
- Где этот код взаимодействует с существующей системой

## O - Operations
**Конкретные шаги реализации:**

### Этап 1: [Название этапа]
1. [ ] Шаг 1.1 — описание
2. [ ] Шаг 1.2 — описание
3. [ ] Шаг 1.3 — описание

**Ожидаемый результат этапа:** [что будет готово]

### Этап 2: [Название этапа]
1. [ ] Шаг 2.1 — описание
2. [ ] Шаг 2.2 — описание

**Ожидаемый результат этапа:** [что будет готово]

### Этап 3: [Название этапа]
[аналогично...]

## N - Norms
**Инженерные стандарты:**

### Именование
- Функции: verbNoun (например, `createUser`)
- Переменные: descriptiveCamelCase (например, `userProfile`)
- Константы: UPPER_SNAKE_CASE
- Компоненты: PascalCase

### Код
- Максимальная длина функции: 50 строк
- Обязательная валидация входных данных
- Defensive programming: предположи, что входные данные могут быть некорректны

### Наблюдаемость
- Логирование: INFO для бизнес-событий, ERROR для исключений
- Метрики: ключевые показатели производительности
- Трассировка: correlation ID для запросов

### Тестирование
- Покрытие: минимум 80% для нового кода
- Типы тестов: unit, integration, e2e
- Именованные тесты: должно быть понятно что тестируется

### Документация
- JSDoc для всех публичных API
- Примеры использования в комментариях

## S - Safeguards
**Неизменные границы:**

### Безопасность
- Никаких секрета в коде
- Валидация всех пользовательских вводов
- Rate limiting: [значения]

### Производительность
- Время ответа API: < 200ms для 95-го перцентиля
- Память: < 500MB для процесса
- Конкурентность: максимум [N] одновременных запросов

### Инварианты
- [ ] Никогда не удалять данные без архивации
- [ ] Транзакционная целостность
- [ ] Idempotency для критических операций

### Ограничения
- Совместимость: backward compatible с v[версия]
- Браузеры: Chrome, Firefox, Safari (последние 2 версии)
- Node.js: LTS версии только

---


## Проверка качества Canvas

Перед генерацией кода проверь:
- [ ] Все требования четко определены
- [ ] Сущности и отношения ясны
- [ ] Подход обоснован
- [ ] Структура соответствует архитектуре проекта
- [ ] Operations конкретны и тестируемы
- [ ] Norms соответствуют стандартам команды
- [ ] Safeguards покрывают критические риски
```

## Пример использования

**Вход:** "Нужно добавить аутентификацию пользователей"

**Выход:** REASONS Canvas с:
- R: Требования к аутентификации (регистрация, вход, сброс пароля)
- E: User entity, Session entity, Token entity
- A: JWT-based auth с refresh tokens
- S: Новые маршруты, middleware, сервисы
- O: 4 этапа реализации с конкретными шагами
- N: Стандарты безопасности, логирования, тестирования
- S: Rate limiting, секреты в env vars, timeout constraints

## Когда использовать

- Перед началом реализации новой функции
- При получении крупного требования
- Для согласования подхода перед кодированием
- При онбординге на новую задачу
