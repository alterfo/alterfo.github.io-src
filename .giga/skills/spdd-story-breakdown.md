# SPDD Story Breakdown

Этот скилл помогает разбивать крупные требования на независимые, доставляемые пользовательские историиfollowing INVEST principle.

## INVEST Принцип

**I**ndependent — Независимая  
**N**egotiable — Обсуждаемая  
**V**aluable — Ценная  
**E**stimable — Оцениваемая  
**S**mall — Маленькая  
**T**estable — Тестируемая

## Инструкция по разбиению

### Шаг 1: Понимание требования

```markdown
**Исходное требование:**
[Вставьте полное описание требования]

**Контекст:**
- Бизнес-цель:
- Пользовательская ценность:
- Ограничения:
```

### Шаг 2: Выделение функциональных границ

Определи независимые функциональные границы:
- Какие функции могут быть доставлены отдельно?
- Какие зависимости между функциями?
- Что является MVP?

### Шаг 3: Создание историй

Для каждой функциональной границы создай историю по шаблону:

```markdown
# User Story: [Название]

## Описание
**Как** [роль пользователя]  
**Я хочу** [действие/функция]  
**Чтобы** [ценность/цель]

## Критерии приемки (Given-When-Then)

### Сценарий 1: [Название сценария]
**Given** [исходное состояние]  
**When** [действие пользователя]  
**Then** [ожидаемый результат]

### Сценарий 2: [Название сценария]
**Given** [исходное состояние]  
**When** [действие пользователя]  
**Then** [ожидаемый результат]

### Сценарий 3: [Граничный случай]
**Given** [исходное состояние]  
**When** [действие пользователя]  
**Then** [ожидаемый результат]

## Технические требования
- [ ] Зависимости: [список]
- [ ] API контракты: [описание]
- [ ] База данных: [изменения]
- [ ] Конфигурация: [новые настройки]

## Оценка сложности
**Story Points:** [1/2/3/5/8/13]
**Риски:** [описание]

## Definition of Done
- [ ] Код написан и рецензирован
- [ ] Тесты написаны и проходят
- [ ] Документация обновлена
- [ ] Нет регрессии
- [ ] Deployed to staging
```

### Шаг 4: Проверка INVEST

Для каждой истории проверь:

| Критерий | Вопрос | Да/Нет |
|----------|--------|--------|
| **Independent** | Можно ли доставить эту историю без других? | |
| **Negotiable** | Можно ли обсудить детали реализации? | |
| **Valuable** | Понятна ли бизнес-ценность? | |
| **Estimable** | Можно ли оценить усилия? | |
| **Small** | Можно ли доставить за спринт? | |
| **Testable** | Есть ли четкие критерии тестирования? | |

## Пример

**Исходное требование:**
"Система должна позволять пользователям управлять своими заказами, включая создание, просмотр, редактирование и отмену заказов"

**Разбиение на истории:**

### Story 1: Создание заказа
- **Independent:** Да, не зависит от других
- **Valuable:** Пользователь может сделать первый заказ
- **Small:** Только создание, без редактирования
- **Testable:** Given-When-Then сценарии для создания

### Story 2: Просмотр списка заказов
- **Independent:** Да, может быть доставлена отдельно
- **Valuable:** Пользователь видит историю заказов
- **Small:** Только просмотр, без фильтрации
- **Testable:** Сценарии для пустого/заполненного списка

### Story 3: Детали заказа
- **Independent:** Зависит от Story 1 и 2
- **Valuable:** Пользователь видит полную информацию
- **Small:** Только отображение деталей
- **Testable:** Сценарии для разных статусов заказа

### Story 4: Отмена заказа
- **Independent:** Зависит от создания заказа
- **Valuable:** Пользователь может исправить ошибку
- **Small:** Только отмена, без редактирования
- **Testable:** Сценарии для разных статусов

## Зависимости между историями

```
Story 1 (Создание) → Story 2 (Список) → Story 3 (Детали) → Story 4 (Отмена)
```

## Шаблоны для разных типов задач

### API Endpoint
```markdown
**Как** клиент API  
**Я хочу** вызывать [endpoint]  
**Чтобы** [цель]

**Критерии:**
- POST /api/resource с валидным payload → 201 Created
- POST /api/resource с невалидным payload → 400 Bad Request
- POST /api/resource без auth → 401 Unauthorized
```

### UI Feature
```markdown
**Как** пользователь интерфейса  
**Я хочу** видеть [элемент]  
**Чтобы** [цель]

**Критерии:**
- Элемент отображается при [условии]
- При клике происходит [действие]
- Валидация показывает [сообщение]
```

### Data Migration
```markdown
**Как** администратор системы  
**Я хочу** выполнить миграцию данных  
**Чтобы** [цель]

**Критерии:**
- Миграция обрабатывает [N] записей
- Откат работает при ошибке
- Логирование всех изменений
```

## Советы

1. **Начинай с вертикальных срезов** — каждая история доставляет ценность
2. **Избегай технических историй** — включай технические задачи в функциональные истории
3. **Используй spike для неизвестного** — если невозможно оценить, создай историю исследования
4. **Ограничивай размер** — если история > 8 SP, разбей дальше
5. **Проверяй независимость** — если история требует другой, подумай как сделать независимой
