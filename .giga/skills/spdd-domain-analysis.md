## Шаблон отчета анализа

```markdown
# Domain Analysis Report: [Название функции]

## 1. Доменные ключевые слова

### Сущности
| Сущность | Описание | Отношения |
|----------|----------|-----------|
| User | Пользователь системы | имеет Orders, Profiles |
| Order | Заказ пользователя | содержит Items, принадлежит User |

### Действия
| Действие | Сущность | Результат |
|----------|----------|-----------|
| createOrder | User | Создает новый Order |
| updateOrder | Order | Обновляет статус заказа |

### Бизнес-правила
1. Order не может быть изменен после отправки
2. User может иметь максимум 5 активных Order
3. Price не может быть отрицательным

### Состояния
- Order Status: `DRAFT` → `SUBMITTED` → `PROCESSING` → `SHIPPED` → `DELIVERED`
- Order Status: `DRAFT` → `CANCELLED`

## 2. Анализ кода

### Существующие модели
```typescript
// src/models/User.ts
class User {
  id: string;
  email: string;
  orders: Order[];
}

// src/models/Order.ts
class Order {
  id: string;
  userId: string;
  status: OrderStatus;
  items: OrderItem[];
  createdAt: Date;
}
```

### Паттерны в проекте
- Repository Pattern для доступа к данным
- Service Layer для бизнес-логики
- DTO для передачи данных

### Зависимости
- TypeORM для ORM
- JWT для аутентификации
- Redis для кэширования

## 3. Риски

### Технические
1. **Масштабируемость** — Order может расти до 1M записей — использовать партиционирование
2. **Конкурентность** — обновление Order требует оптимистичной блокировки

### Доменные
1. **Сложность статусов** — требуется четкое определение переходов между статусами

## 4. Направления дизайна

### Рекомендуемый подход
- **CQRS** для разделения команд и запросов
- **Domain Events** для асинхронных процессов
- **State Machine** для управления статусами Order

### Ключевые решения
1. Использовать оптимистичную блокировку для Order
2. Реализовать Domain Events для OrderStatusChange
3. Добавить индексы на userId и status для частых запросов

## 5. Влияние на существующую систему

### Модули для изменения
- `src/services/orderService.ts`
- `src/models/Order.ts`
- `src/routes/orders.ts`

### API контракты
- POST /api/orders — создание
- GET /api/orders/:id — получение
- PATCH /api/orders/:id — обновление

### Миграции БД
- Добавление новых таблиц
- Индексы для производительности

## 6. Рекомендации для Canvas

### На что обратить внимание в Requirements
- Четко определить переходы статусов
- Определить валидацию для каждого статуса

### Для Entities
- Указать все свойства OrderItem
- Определить отношения между сущностями

### Для Approach
- Описать поток создания заказа
- Указать паттерны для использования

### Для Safeguards
- Rate limiting на создание Order
- Валидация цены и количества
- Idempotency для предотвращения дублирования
```

## Инструменты для анализа

# Найти связанные тесты
find src/ -name "*.test.ts" | xargs grep -l "Order"

# Анализ зависимостей
grep -r "import.*Order" src/
```

```bash

tree -L 3 src/

# Счетчик строк по модулям
wc -l src/**/*.ts
```

## Советы для качественного анализа

1. **Пойми домен перед кодом** — говори с предметными экспертами
2. **Ищи паттерны** — что уже работает в проекте
3. **Документируй предположения** — явно фиксируй что не уверен
4. **Проверяй тесты** — они раскрывают ожидаемое поведение
5. **Смотри на миграции** — как эволюционировала схема данных
